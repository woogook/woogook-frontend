# Git Commit Guide

- 특별한 사유가 없으면 `main` 브랜치가 아닌 적절한 작업 브랜치에서 `commit`을 진행한다.
- 문서와 설명 문장은 `한글`로 작성하는 것을 원칙으로 하며, `영어` 표기가 더 정확하거나 자연스러운 경우에는 괄호를 사용해 한글과 영어를 함께 작성한다. 예: 강화학습(RL, Reinforcement Learning)

## `commit`을 진행하기 전에 구체적으로 검토해야 할 항목

### 공통

- `절대 경로`와 같이 특정 개발 환경에 종속된 값이 없는지 검토한다.

### 코드

- 실행 오류, 동작 회귀, 유지보수 리스크를 중심으로 코드 리뷰를 진행한다.
- 필요한 테스트 코드나 다른 검증 방법이 마련되었는지 확인한다.
- 리팩터링할 만한 포인트가 없는지 검토한다.
- `docstring`이 필요한 대상에 누락된 곳이 없는지 확인하고, 기존 `docstring`은 동작 설명이 바뀐 경우에만 수정이 필요한지 검토한다.

### 문서

- `README`, `ADR` 등 문서가 상세하고 정확하게 작성되었는지 검토한다.
- 여러 문서로 나뉜 ADR을 통합할 만한 포인트가 없는지 검토한다. ADR을 통합할 때는 '어떤 문서를 통합했다/흡수했다'와 같은 문구를 추가하지 말고, 내용을 자연스럽게 정리한다.

## `commit`을 진행할 때 규칙

### 제목

- `<prefix>: <작업 내용을 잘 요약한 문장>` 형식으로 작성한다.

### `commit` 본문

- 제목만으로 충분하지 않은 배경이나 변경 이유가 있으면 본문에 적는다.
- 검토 시 알아야 할 비자명한 제약이나 주의사항이 있으면 본문에 함께 적는다.
- 항목이 여러 개일 때는 `bullet list`를 활용할 수 있다.
