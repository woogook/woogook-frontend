# Review Workflow

## 목적

- implementer 관점이 아니라 third-party reviewer 관점에서 작업 결과를 검토한다.
- 발견된 문제를 조치하거나 남은 리스크를 후속 조치 가능한 위치에 기록한다.

## 입력

- 변경된 코드와 문서
- 기준 GitHub Issue
- 테스트 및 검증 결과

## 실행 순서

1. 변경된 파일만 보지 말고 diff, 관련 코드, 호출 흐름, 기존 구현 패턴, 테스트, 설정, 문서 영향을 함께 확인한다.
2. 외부 리뷰어가 지적할 만한 항목을 먼저 점검한다.
   - 정확성
   - 예외/경계 조건
   - 부작용
   - 가독성
   - 일관성
   - 성능
   - 보안
   - 프로젝트 컨벤션 적합성
3. 코드가 동작하는지만 보지 말고, 현재 방식이 안전한지와 기존 코드베이스 맥락에 맞는지도 확인한다.
4. 문제를 발견하면 수정하거나, 바로 수정할 수 없으면 후속 조치가 가능한 위치에 남긴다.

## 완료 조건

- 리뷰 관점에서 중요한 문제를 확인했다.
- 발견된 문제는 조치되었거나 후속 조치가 가능한 형태로 기록되었다.
